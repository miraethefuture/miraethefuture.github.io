# 커밋 메시지 가이드

이 레포지토리는 블로그 콘텐츠와 UI 코드가 함께 관리되므로, 커밋 메시지는 변경 성격을 빠르게 구분할 수 있어야 합니다.

## 기본 형식

```text
type: 변경 요약
```

예시:

```text
feat: 문서형 블로그 레이아웃 개편
content: LottoMate 폴더 구조 리팩토링 포스트 추가
style: 본문 타이포그래피와 색상 정리
fix: 모바일 사이드바 레이아웃 깨짐 수정
```

## 권장 타입

### `feat`

- 사용자에게 보이는 새 기능, 새 페이지, 새 탐색 구조 추가
- 예: 검색 기능 추가, 태그 페이지 추가, 레이아웃 구조 개편

### `content`

- 블로그 포스트, About 문구, 소개 카피, 문서 텍스트 변경
- 예: 새 글 작성, 기존 포스트 내용 수정

### `docs`

- 저장소 운영 문서, 가이드, 규칙 문서 변경
- 예: 커밋 메시지 규칙 추가, 작업 가이드 업데이트

### `style`

- 기능 변화 없이 시각 디자인만 바뀌는 변경
- 예: 색상, 간격, 폰트, 타이포그래피 조정

### `fix`

- 버그, 잘못된 링크, 레이아웃 깨짐, 잘못된 메타데이터 수정
- 예: 모바일에서 본문 폭 깨짐 수정

### `refactor`

- 동작 변화 없이 코드 구조만 정리하는 변경
- 예: 컴포넌트 분리, 유틸 함수 정리, 중복 제거

### `chore`

- 설정, 의존성, 빌드, 배포, 도구 관련 작업
- 예: lint 설정 추가, package 정리, analytics 설정

## 작성 원칙

- 메시지는 한글로 작성합니다.
- 첫 줄은 짧고 구체적으로 씁니다.
- 하나의 커밋에는 하나의 목적만 담는 것을 권장합니다.
- 콘텐츠 변경과 UI 변경이 함께 있으면 가능하면 커밋을 분리합니다.

## 추천 예시

```text
feat: 태그 아카이브 페이지 추가
content: iOS 아키텍처 리팩토링 포스트 추가
docs: 커밋 메시지 작성 가이드 추가
style: 상세 페이지 본문 가독성 개선
fix: 검색 페이지 입력창 포커스 스타일 수정
refactor: 문서 셸 컴포넌트 구조 정리
chore: Next.js 빌드 설정 업데이트
```
