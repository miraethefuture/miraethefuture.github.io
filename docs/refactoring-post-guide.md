# 리팩토링 기록 포스트 작성 가이드

이 문서는 프로젝트 리팩토링 기록 포스트를 외부에서 작성한 뒤 이 블로그에 옮길 때 바로 사용할 수 있도록 만든 가이드입니다.

## 1. 파일 위치와 이름

- 위치: `_posts/`
- 파일명 형식: `YYYY-MM-DD-slug.md`
- 예시: `2026-03-10-auth-flow-refactoring.md`

`slug`는 영문 소문자와 하이픈(`-`) 조합을 권장합니다.

## 2. Front matter 규칙

현재 앱에서 실질적으로 사용하는 핵심 필드는 아래입니다.

- `title`: 포스트 제목
- `date`: 발행일 (`YYYY-MM-DD`)
- `summary`: 목록/상세 상단에 노출될 짧은 요약
- `tags`: 태그 배열
- `type`: `project` 또는 `til`

추가로 아래 필드도 넣을 수 있습니다.

- `featured`: 메인 노출 여부
- `links`: 관련 링크

권장 예시는 아래와 같습니다.

```yaml
---
title: 프로젝트 리팩토링 기록 - 인증 흐름 정리
date: 2026-03-10
summary: 로그인 상태 처리와 네트워크 계층 책임을 분리하면서 인증 흐름을 단순화한 과정을 정리했다.
tags:
  - iOS
  - Refactoring
  - Architecture
type: project
featured: false
links:
  github: https://github.com/username/repo
  note: https://example.com/reference
---
```

## 3. 작성 원칙

- 문제와 해결을 같이 적습니다. "무엇을 바꿨는지"보다 "왜 바꿨는지"가 먼저 보여야 합니다.
- 변경 전/후를 비교 가능하게 적습니다. 가능하면 코드, 구조도, 책임 분리 기준을 함께 남깁니다.
- 결과를 과장하지 않습니다. 체감 개선인지, 측정된 개선인지 구분해서 씁니다.
- 한 포스트에는 하나의 큰 리팩토링 주제를 담는 편이 읽기 쉽습니다.
- 회고를 남깁니다. 실패한 시도나 판단 기준도 기록 가치가 있습니다.

## 4. 권장 본문 구조

아래 순서를 기본 템플릿으로 쓰면 됩니다.

### 1) 배경

- 어떤 기능/화면/모듈을 다뤘는지
- 언제 리팩토링이 필요하다고 느꼈는지
- 작업 범위를 어디까지 잡았는지

### 2) 문제 정의

- 기존 코드의 불편함
- 유지보수, 확장성, 테스트, 성능, 가독성 중 무엇이 문제였는지
- 실제로 겪은 증상이나 사례

### 3) 목표

- 이번 리팩토링에서 해결하려는 것
- 일부러 하지 않은 것

### 4) 기존 구조

- 기존 코드 또는 구조 설명
- 왜 문제가 생기기 쉬운 구조였는지

### 5) 진행 과정

- 어떤 순서로 바꿨는지
- 중간에 시도했다가 버린 방법
- 트레이드오프

### 6) 변경 후 구조

- 핵심 변경 사항
- 역할 분리 기준
- 전/후 비교 포인트

### 7) 결과

- 코드 길이 감소, 중복 제거, 의존성 정리, 테스트 용이성 등
- 수치가 있으면 수치 포함

### 8) 배운 점

- 다음에 비슷한 작업을 할 때 적용할 원칙
- 아쉬웠던 점

## 5. 복붙용 템플릿

```md
---
title: 프로젝트 리팩토링 기록 - [주제]
date: 2026-03-10
summary: [이번 리팩토링의 핵심을 1~2문장으로 요약]
tags:
  - [기술스택]
  - Refactoring
  - [주제]
type: project
featured: false
links:
  github: [선택]
  note: [선택]
---

- 기존 코드의 문제를 구체적으로 적습니다.
- 가능하면 "불편했다"보다 "어떤 변경에서 비용이 커졌다"처럼 씁니다.

- 이번 작업에서 해결하려는 범위를 적습니다.
- 이번 포스트에서 다루지 않는 범위도 함께 적으면 좋습니다.

## 진행 과정

### 1. 첫 번째 변경

- 무엇을 바꿨는지
- 왜 그렇게 바꿨는지

### 2. 시도했다가 버린 방법

- 실패한 접근이 있다면 짧게 남깁니다.

## 결과

- 가독성
- 테스트 편의성

## 6. 제목 작성 팁

아래 형식을 추천합니다.

- `프로젝트 리팩토링 기록 - 인증 흐름 단순화`
- `프로젝트 리팩토링 기록 - 네트워크 레이어 역할 분리`
- `프로젝트 리팩토링 기록 - 중복 UI 상태 처리 제거`

너무 넓은 제목보다는 "무엇을 어떻게 개선했는지"가 드러나는 제목이 좋습니다.

## 7. 태그 작성 팁

태그는 3~5개 정도가 적당합니다.

- 기술/프레임워크: `iOS`, `Swift`, `RxSwift`, `React`
- 작업 성격: `Refactoring`, `Architecture`, `Performance`
- 도메인/주제: `Login`, `Networking`, `State`

## 8. 옮겨올 때 체크리스트

- 파일명이 `YYYY-MM-DD-slug.md` 형식인지 확인
- `summary`를 작성했는지 확인
- `tags`가 배열 형태인지 확인
- `type`이 `project` 또는 `til`인지 확인
- 코드 블록 언어가 지정되어 있는지 확인
- 문서 안의 임시 메모나 초안 문구를 제거했는지 확인

## 9. 이 블로그 기준 추천

프로젝트 리팩토링 기록이라면 기본값은 아래를 권장합니다.

- `type: project`
- `tags`에 `Refactoring` 포함
- `summary`는 120자 안팎으로 짧고 명확하게 작성
